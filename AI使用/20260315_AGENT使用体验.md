# 20260315 AGENT使用体验

## Agent 中直接调用 Skill vs 描述目标的差异

> 不要直接调用命令 而是告知agent想要使用的命令

### 问题背景

在使用 Cursor Agent 时，直接调用 Skill 生成的内容相对死板、不达预期，而描述目标让 Agent 自行决策时效果更好。这背后存在本质性的行为差异。

### 两种方式的核心区别

#### 直接调用 Skill（"用 XX skill"）

当直接说"用监控 skill"或"按照 XX skill 来做"时，Agent 的行为模式是：

- **Skill 本身变成了目标** —— Agent 会把 SKILL.md 当作"指令清单"逐步执行
- 倾向于**机械地套用模板**，因为它认为你要的就是"按 skill 里写的来"
- 对上下文的理解和适配会弱化，因为 Agent 的注意力集中在"遵循 skill 的步骤"上
- 结果就是：**输出死板、模板感强**

#### 描述目标（"我想实现 XX"）

当描述目标时，Agent 的行为模式是：

- **目标是业务需求**，Skill 只是实现手段之一
- Agent 会先理解场景、探索代码库、分析上下文
- 然后**选择性地**应用 skill 中的相关部分，并根据实际情况做适配
- 有更大的自由度去组合多个 skill、跳过不适用的部分、补充 skill 未覆盖的内容
- 结果通常**更贴合实际需求**

### 根本原因

本质上是 **"指令遵循"和"目标导向"两种推理模式的差异**：


|           | 直接调用 Skill  | 描述目标       |
| --------- | ----------- | ---------- |
| Agent 视角  | "我要按这个文档执行" | "我要解决这个问题" |
| Skill 的角色 | 目标本身        | 工具/参考      |
| 灵活度       | 低，逐步执行      | 高，按需取用     |
| 上下文融合     | 弱           | 强          |


### 最佳实践

**推荐方式：描述目标 + 提及 skill 作为参考**

与其说：

> "用 backend-monitor-skill 给我的服务加监控"

不如说：

> "我的 XX 服务需要加监控，核心关注订单成功率和响应延迟。可以参考 @backend-monitor-skill 的规范"

这样 Agent 会：

1. 以业务目标为导向
2. 把 skill 当作质量和规范的参考
3. 根据具体场景做适配

### Skill 编写优化建议

如果某个 skill 产出总是过于死板，可能是 **skill 本身写得太过程序化**（步骤过细、模板过多、缺少"根据场景判断"的指引）。优化方向：

- 增加更多"判断逻辑"而非纯模板
- 给 Agent 留出适配空间
- 明确哪些是硬性规范、哪些是可灵活调整的参考

## Agent合规性缺失

> 即使在沟通中明确强调了要求，并且内容也被写入了规格文件，Agent 在执行时仍然不遵守——这不是偶发现象，而是当前 LLM 的结构性问题。

### 什么是 Agent 合规性缺失

Agent 合规性缺失是指：你已经通过对话、规格文档、项目原则等方式明确提出了约束，但 Agent 在实际执行时未能遵守。业界研究表明，Agent 对明确指令的遵循率通常只有 **70-90%**——这不是个例，而是当前 LLM 的普遍现象。

### 根因分析

#### 1. 指令优先级竞争

Agent 同时受到多层指令约束，它们在上下文中争夺注意力：


| 层级           | 来源                     | 实际优先级  |
| ------------ | ---------------------- | ------ |
| 系统提示词        | Agent 平台内置             | 最高     |
| 命令文件         | 如 implement.md 的步骤     | 高      |
| 用户对话         | 你在聊天中说的要求              | 中高     |
| 规格文档         | spec.md / plan.md 中的约束 | **中低** |
| Constitution | 项目原则                   | 低      |


当你的要求写在 spec.md 里但与 Agent 的"默认行为模式"（训练时学到的通用模式）冲突时，Agent 倾向于用通用模式。

#### 2. 多步转换中的信息衰减

Spec Kit 的工作流是一条链：

```
spec.md → plan.md → tasks.md → 实现代码
```

每一步转换都可能丢失信息。spec.md 中的一个约束，在 plan.md 中可能被概括，在 tasks.md 中被进一步简化，到实现时已经完全丢失。

#### 3. 自然语言的有损性

Markdown 天然是有损媒介。你觉得写得清楚的约束，对 LLM 来说可能存在多种解读。而且当 Agent 生成到第 N 个文件时，早期读到的那条规则在"工作记忆"中已经衰减，它会退回到训练数据中最常见的模式。

#### 4. 上下文中的注意力稀释

当 Agent 在实现阶段同时加载命令文件、spec、plan、tasks、constitution、代码文件和对话消息时，你的关键约束只是海量 token 中的几行文字。上下文越长，单条约束分到的注意力权重越低（第 3 节从架构层面详述这一机制）。

#### 5. "最后消息"偏差

LLM 天然对序列末端的内容给予更多注意力。如果你在对话早期提出了一个要求，后续又说了很多别的内容，那个早期要求被遵守的概率会大幅降低。这也是为什么"在对话末尾重申关键要求"是有效策略（第 3 节给出具体做法）。

关键动作：通过外部约束告知agent需要考虑的上下文，补充相关信息

### 应对方向

合规性缺失需要从两个层面同时入手：

- **注意力机制层**：让约束在上下文中更"响亮"——通过定向重复，把关键约束放到 Agent 注意力最集中的位置（下一节详述）
- **推理模式层**：让约束成为 Agent 目标的一部分，而非背景信息——即第 1 节讨论的"描述目标"方式

## 为什么 AI Agent 总是"忘记"你的要求？—— Prompt Repetition 论文的启示

上一节分析了 Agent 合规性缺失的三个根因。那么在 LLM 的架构层面，到底是什么机制导致了这些问题？

### 不是 Agent "不听话"，而是它"看不清"

Google Research 2025 年 12 月的论文 [Prompt Repetition Improves Non-Reasoning LLMs](https://arxiv.org/abs/2512.14982) 揭示了一个关键事实：

> 把 prompt 原样重复一遍，就能让非推理模式下 LLM 的准确率大幅提升——某些任务从 21% 飙到 97%，且不增加延迟和生成成本。

这背后的原因是 LLM 的**因果注意力机制**：模型在处理文本时是从左到右的，**前面的内容无法"看到"后面的内容**。当你的关键要求出现在一大段上下文的前部，而 Agent 生成代码在序列末尾时，那些要求的信号已经大幅衰减。

简单说：**不是 Agent 不想遵守你的要求，而是你的要求被淹没在海量上下文中，Agent 在生成代码的时候已经"看不清"了。**

### Agent 实际"看到"的是什么

当 Agent 执行一个实现任务时，它的上下文窗口大致是这样的序列：

```
[系统提示词] → [命令文件] → [spec.md] → [plan.md] → [tasks.md]
→ [constitution.md] → [当前代码文件] → ... → [你的对话消息] → 【生成代码】
```

你的关键约束可能在 `spec.md` 里，但它距离实际生成代码的位置隔了几千甚至几万个 token。在因果注意力下：

- **位置越靠前的内容，对后续生成的影响力越弱**
- **上下文越长，单条约束获得的注意力权重越低**
- **模型天然偏向关注最近的内容**

这就是为什么你在 spec 里写了"所有 API 用 snake_case"，Agent 在生成第 15 个 endpoint 时却用了 camelCase。

### 实践建议：让 Agent "听到"你的要求

基于论文原理，核心思路是：**让关键约束出现在 Agent 注意力最集中的位置，并通过重复强化信号。**

#### 1. 在任务描述中内嵌约束，而非仅在 spec 中写一次

```markdown
Task 3.2: 实现用户 API endpoint
- 创建 GET /api/users endpoint
- **约束：所有响应字段必须使用 snake_case 命名**
- **约束：错误响应必须遵循 RFC 7807 格式**
```

约束紧邻执行点，Agent 在处理这个任务时能直接"看到"。

#### 2. 在对话消息的末尾重申关键要求

利用模型对序列末端的注意力偏好：

```
请实现 Task 3.2。

提醒：所有字段必须使用 snake_case，错误格式遵循 RFC 7807。
```

最后一句话是 Agent 在开始生成前"最后看到"的内容，遵守概率最高。

#### 3. 缩小单次任务的上下文范围

每次只给 Agent 少量、聚焦的任务和约束。上下文越短，你的约束在其中的"浓度"越高，被遵守的概率越大。避免一次性让 Agent 加载整个规格体系后执行大量任务。

#### 4. 用 Checklist 实现结构化的"重复"

将关键约束转化为检查清单，在实现前后逐项核查。Checklist 本质上是让 Agent 再"读一遍"你的要求，等同于论文中 prompt repetition 的结构化实现。

#### 5. 接受"验证-修正"循环是必要环节

当前技术条件下，100% 的规格合规率是不现实的预期。更务实的做法是：

- 实现完成后，主动对照规格逐条验收
- 发现违规时明确指出"你违反了 spec 中第 X 条要求"
- 让 Agent 针对性修正，而非期望一次性完美

### 局限性：呐喊不是万能的

论文的实验场景是**单轮、短上下文、非推理任务**。而 Agent 写代码是**多轮、长上下文、推理任务**。论文自己的数据显示：推理模式下效果从 47/70 胜降到 5/28 胜。

所以简单地把规格要求复制三遍塞进 prompt，不会复现 21% → 97% 的奇迹。论文的价值在于揭示了**原理**（因果注意力下位置和重复影响关注度），而非提供一个可以机械套用的操作。在 Agent 编码场景下，需要将"呐喊"从粗暴的全文重复升级为**定向重复**——在关键决策点附近放置约束。

### 一句话总结

> **如果你希望 Agent 遵守某个要求，就让这个要求在它的上下文中足够"响亮"——靠近执行点、反复出现、在最后一句话中重申。**

## 规格隧道视野

相关讨论： [SpecKit上下文忽略问题分析与解法](https://zzzzzr.github.io/AI%E4%BD%BF%E7%94%A8/20260206_SpecKit%E4%B8%8A%E4%B8%8B%E6%96%87%E5%BF%BD%E7%95%A5%E9%97%AE%E9%A2%98%E5%88%86%E6%9E%90%E4%B8%8E%E8%A7%A3%E6%B3%95/)

### 你可能会遇到的场景

当你花了两天写了一份详尽的 spec，实现过程中发现某个设计不合理，却不愿意改 spec 而是强行按原方案来；或者产品需求已经变了，你仍然按旧 spec 执行，因为"spec 还没更新"——这时候问题可能不在 Agent，而在你自己。

### 什么是规格隧道视野

Spec-Driven Development（SDD）中的一种反模式。指团队过度依赖和锁定在静态规格文档上，忽略了软件开发本质上是探索性、迭代性过程。就像在隧道中看东西——只盯着规格文档，看不到规格之外的更大画面。这不是 Spec Kit 独有的问题，而是所有"规格先行"方法论的通病。

### 与 Agent 合规性缺失的区别

这两个问题方向相反：


| 问题              | 主体       | 方向                    |
| --------------- | -------- | --------------------- |
| **规格隧道视野**      | 人/团队     | 对规格**太执着**，忽略规格之外的现实  |
| **Agent 合规性缺失** | AI Agent | 对规格**不够关注**，忽略规格之内的约束 |


前者是人的认知陷阱，后者是模型架构的固有缺陷。

### 深层表现

规格隧道视野一旦出现，通常有两个不易察觉的认知状态：

- **虚假的完整感**：详细的规格文档给团队一种"所有情况都已覆盖"的错觉，减少了迭代和探索，使开发变成脆弱的类瀑布流程
- **错误的抽象层级**：规格过度关注实现细节（字段定义、Schema、函数签名），而忽略了真正重要的东西——意图、约束和上下文

### 什么时候会出现

规格隧道视野通常不会在使用 Spec Kit 的早期出现，而是在你**越来越熟练、规格写得越来越详尽**之后才需要警惕：


| 场景       | 表现                                  |
| -------- | ----------------------------------- |
| 规格与实现矛盾时 | 代码跑起来发现设计不合理，但因为"规格是花了力气写的"而强行按规格来  |
| 需求变化时    | 产品需求变了，仍按旧 spec 执行，因为"spec 还没更新"    |
| 团队协作时    | 团队成员有不同理解，但因为"规格里写了"而不再讨论           |
| 过度追求规格完美 | 花大量时间完善 spec，迟迟不进入实现，试图在纸面上消除所有不确定性 |


**判断标准**：如果你发现自己在说"spec 里就是这么写的，按 spec 来"而不是"这个 spec 写得对吗"，那就是隧道视野了。

### 如何避免

- 把规格当作"活契约"而非"完美蓝图"——实现中发现不合理就改规格
- 用 `[NEEDS CLARIFICATION]` 标记未知点，不假装所有问题都已有答案
- 定期回顾：当前规格是否仍然反映实际需求？有没有过时的约束？
- 利用 Spec Kit 的 `/speckit.clarify` 命令主动发现规格中的模糊点

Spec Kit 还内置了几个对抗隧道视野的机制：

- **持续细化（Continuous Refinement）**：AI 持续分析规格中的歧义、矛盾和遗漏，作为不间断的过程而非一次性关卡
- **双向反馈（Bidirectional Feedback）**：生产环境的现实反馈（指标、事故、运营经验）回流到规格的演进，规格不是单向驱动代码
- **探索性分支（Branching for Exploration）**：从同一规格生成多种实现方案，探索不同优化方向，避免锁定在单一方案

## 回顾：各话题的关联与组合策略

读完前面四个独立话题后，可以看到它们之间存在互补关系。

### 四个话题解决不同层面的问题


| 话题                     | 层面   | 核心问题             |
| ---------------------- | ---- | ---------------- |
| Skill vs 描述目标          | 操作方式 | 怎么跟 Agent 说话效果更好 |
| Agent 合规性缺失            | 问题诊断 | 为什么 Agent 不遵守要求  |
| Prompt Repetition / 呐喊 | 底层机制 | 因果注意力如何导致信息丢失    |
| 规格隧道视野                 | 认知反思 | 人自身是否被规格束缚       |


### 两种缓解合规性的方法

第 1 节和第 3 节分别从不同层面缓解第 2 节提出的合规性问题：


| 方法             | 作用层面   | 效果                    |
| -------------- | ------ | --------------------- |
| 描述目标而非调用 Skill | 推理模式层  | 让约束成为 Agent 的目标而非背景信息 |
| 定向重复 / 呐喊      | 注意力机制层 | 让约束信号在上下文中更"响亮"       |


### 组合策略示例

同时运用两种方法：

```
我的订单服务需要实现退款功能，核心需求是退款流程和状态管理。
请参考 spec.md 中的技术方案。

硬性要求：
- 每个 public 方法必须有对应单测
- 单测必须遵循 AAA 模式（Arrange-Act-Assert）
- 测试覆盖率不低于 80%
```

这里：描述目标（退款功能）→ 激活目标导向推理；末尾列出硬性要求 → 放在注意力焦点位置。
