# Spec Kit 上下文忽略问题分析与解法

**日期**: 2026-02-06

## 问题描述

使用 Spec Kit 进行规格驱动开发（SDD）时，AI Agent 容易陷入"规格隧道视野"——过于关注 `spec.md`、`plan.md`、`tasks.md` 这些规格文件，而忽略了项目的实际上下文。

## 问题根因

AI Agent 在执行 `/speckit.implement` 等阶段时，忽略了以下关键上下文：

1. **已有的代码结构和模式** —— 项目中已有的设计模式、命名约定、架构风格
2. **真实的运行时上下文** —— 依赖关系、配置、环境
3. **跨功能的一致性** —— 与其他已实现功能的代码风格一致性

## Spec Kit 现有的上下文机制

Spec Kit 已经提供了一些上下文传递机制，但可能没有被充分利用：

| 机制 | 文件位置 | 作用 |
|------|----------|------|
| Constitution | `.specify/memory/constitution.md` | 项目级治理原则，在 plan 阶段被校验 |
| Agent Context 文件 | `CLAUDE.md` / `.cursor/rules/specify-rules.mdc` 等 | 项目级指令，每次对话都被读取 |
| update-agent-context 脚本 | `scripts/bash/update-agent-context.sh` | 从 plan.md 自动提取技术栈更新到 Agent Context |
| 跨制品引用 | plan → spec, tasks → plan | 各阶段文件相互引用 |

**上下文传递流程：**

```
User Input → /speckit.specify
    ↓
spec.md (功能级规格)
    ↓
/speckit.plan
    ↓
plan.md + research.md + data-model.md + contracts/
    ↓
update-agent-context.sh (更新项目级 Agent 文件)
    ↓
CLAUDE.md / .cursor/rules/ 等 (项目级上下文)
    ↓
/speckit.tasks → tasks.md
    ↓
/speckit.implement → 代码实现
```

## 解法

### 解法 1：充实 Constitution 文件（全局生效，优先级最高）

`.specify/memory/constitution.md` 是 Spec Kit 提供的项目级记忆机制。在 `/speckit.plan` 阶段会被读取并校验合规性。

**建议添加内容：**

```markdown
## 代码上下文规则

- 在实现任何新功能前，必须先阅读相关模块的现有代码
- 遵循项目中已有的设计模式（列出关键模式）
- 所有新代码必须与现有代码风格保持一致

## 项目架构概览

- src/xxx/ - 核心模块，使用了 xxx 模式
- src/yyy/ - 服务层，统一使用依赖注入
- ...

## 技术约束

- 优先复用已有依赖，不要引入重复功能的新库
- 遵循现有的错误处理模式
- 测试代码应参考同模块已有测试的风格
```

**优点**：一次配置，所有后续功能都会受益；在 plan 阶段就被消费，影响后续所有阶段。

### 解法 2：增强 Agent Context 文件（全局生效，优先级高）

Agent 上下文文件（如 Cursor 的 `.cursor/rules/specify-rules.mdc`）是**每次对话都会被读取**的项目级指令。

可以在文件中手动添加更具体的上下文指引：

```markdown
## 实现时的上下文要求

在执行 speckit implement 时，必须：
1. 先通读目标模块的现有代码，理解当前架构
2. 查看相邻模块的接口和数据流
3. 检查 package.json / requirements.txt 中已有的依赖，优先复用
4. 参考最近 3-5 个 commit 的代码风格
```

**注意**：`update-agent-context.sh` 脚本在更新时会**保留你手动添加的内容**（在标记区域之外的部分），所以自定义内容不会丢失。

### 解法 3：利用 Cursor Rules 机制（针对特定文件/目录自动触发）

在 `.cursor/rules/` 下创建额外的规则文件，例如 `context-awareness.mdc`：

```markdown
---
description: "确保AI在实现功能时关注项目上下文"
globs: ["src/**"]
---

在修改或创建 src/ 下的任何文件时：
1. 先阅读同目录下的其他文件，理解模块结构
2. 检查该模块的 index/barrel 文件，了解导出模式
3. 查看相关测试文件，理解测试约定
4. 新代码必须与现有代码风格一致
```

**优点**：基于 glob 匹配自动触发，不需要每次手动提醒；可以针对不同目录设置不同规则。

### 解法 4：在 Plan 阶段主动锚定上下文

在使用 `/speckit.plan` 时，在交互中主动要求 AI 分析现有代码：

> "在制定计划前，请先分析 `src/services/` 和 `src/models/` 目录下的现有代码结构，确保计划与现有架构一致。"

这样生成的 `plan.md` 会包含对现有代码的引用，后续阶段自然会继承这些上下文。

### 解法 5：在 Implement 阶段手动补充上下文（临时方案）

当感觉 Agent 忽略了上下文，可以在实现过程中直接补充：

> "在实现这个任务前，请先阅读 `src/xxx/existing-module.ts`，新代码需要遵循同样的模式。"

**缺点**：需要每次手动操作，不可复用。

## 解法优先级总结

| 优先级 | 方法 | 效果 | 维护成本 |
|--------|------|------|----------|
| **高** | 充实 Constitution 文件 | 全局生效，影响 plan 及后续所有阶段 | 低（一次配置） |
| **高** | 增强 Agent Context 文件 | 全局生效，每次对话都被读取 | 低（一次配置） |
| **高** | 创建 Cursor Rules 文件 | 针对特定文件/目录自动触发 | 低（一次配置） |
| **中** | Plan 阶段锚定上下文 | 让规格文件本身包含上下文引用 | 中（每个功能需手动） |
| **低** | Implement 时手动补充 | 临时解决特定问题 | 高（每次都需手动） |

## 推荐实施步骤

1. **立即**：完善 `.specify/memory/constitution.md`，加入项目架构、代码风格、技术约束等内容
2. **立即**：在 Agent Context 文件中添加"实现时上下文要求"段落
3. **可选**：为常改动的目录创建 `.cursor/rules/` 下的专属规则文件
4. **养成习惯**：在 `/speckit.plan` 阶段主动引导 AI 阅读现有代码

## 关键洞察

Spec Kit 的设计本身是支持上下文传递的，问题在于：
- **Constitution 文件往往初始化后没有认真填写**，缺少项目级的架构和风格约束
- **Agent Context 文件主要靠脚本自动更新**，缺少手动补充的项目特有指引
- **规格文件（spec/plan/tasks）聚焦于"做什么"**，缺少"参考什么现有代码"的指引

解决这个问题的本质是：**在 AI 接触到的各个上下文入口，都植入"先看现有代码"的指令**。
