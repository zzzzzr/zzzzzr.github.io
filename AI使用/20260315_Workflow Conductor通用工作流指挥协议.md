# Workflow Conductor — 通用工作流指挥协议

## 快速介绍

workflow-conductor 是一条 Cursor 全局 Rule，用于在多步骤任务中维护核心目标、追踪执行进度、持久化关键约束。安装后自动生效，不需要修改任何现有的命令或 skill。

这条 Rule 要解决的是两个在日常使用 agent 时反复出现的问题：对话长了之后早期的约束会被"忘掉"，以及执行多步骤任务时 agent 要么死板逐条跑、要么自作主张跳过某些步骤。核心思路是用一个持久化文件（.cursor/sessions/objective.md）充当"会话置顶消息"，agent 每轮对话前自动读取，保证目标和约束始终在注意力范围内。


## 背景：两个实际问题

在使用 AI Agent（Cursor）进行开发时，我们遇到了两个反复出现的问题：

### 问题 1：多步骤任务的编排困境

当一个任务需要按顺序执行多个命令、调用多个 skill 或 MCP 时，agent 的表现往往走两个极端：

- 死板地逐条执行命令，完全不理解为什么要做这些步骤
- 过于"聪明"，自行判断某个步骤不需要而跳过

我们希望 agent 能聚焦于业务目标，灵活决定每一步怎么做，但不能自作主张跳过某个步骤——因为每一步的产出都是业务知识库的一部分，即使看起来冗余也有保留价值。

### 问题 2：长对话中的注意力分散

在对话的第 3 轮强调过的约束，到第 15 轮时 agent 经常"忘了"。这是大模型注意力机制的固有特性——早期的信息会随着上下文增长而权重下降。以前的应对方式是反复"呐喊"（手动重复关键要求），但这不可持续。

## Cursor 中能约束 agent 行为的机制

在讨论方案之前，先了解 Cursor 中所有能影响 agent 行为的机制，按约束力从高到低排列：

| 层级 | 机制 | 谁控制 | 加载方式 | 约束力 |
| --- | --- | --- | --- | --- |
| 1 | System Prompt（系统提示词） | Cursor 官方 | 始终加载，用户不可修改 | 最强，但用户无法利用 |
| 2 | User Rules（用户规则） | 用户在 Cursor 设置中配置 | 始终加载，全局生效 | 很强，但是静态内容 |
| 3 | Workspace Rules（`.cursor/rules/`） | 用户/团队 | `alwaysApply: true` 始终加载，或按 glob 条件加载 | 强，用户可控的最高层级 |
| 4 | AGENTS.md | 项目维护者 | 项目级始终加载 | 与 Rules 类似，但只有一个文件 |
| 5 | MCP Server Instructions | MCP 服务提供者 | MCP 连接时自动加载 | 中高 |
| 6 | Skills | 用户/团队 | 通过 description 关键词匹配被动触发 | 中，不一定被触发 |
| 7 | Commands（命令正文） | 用户/团队 | 用户显式调用时加载 | 执行期间较强，但作用域限于本次执行 |
| 8 | 对话历史 | 用户 | 自然累积 | 最弱，越早的内容权重越低，长对话会被压缩 |

关键认知：**所有层级的约束最终都是 prompt 的一部分**，通过注意力机制影响 agent 的行为，没有任何一层能做到程序级的"硬约束"。层级越高意味着权重越高、合规率越高，但不是 100%。

补充说明：Cursor 没有 Constitution 文件机制。部分 agent 框架（如 Claude Code）有专门的 constitution/system prompt 覆盖能力，但 Cursor 中最接近的等价物就是 `alwaysApply: true` 的 Workspace Rule。

## 现有机制能否独立解决这两个问题

我们逐一检查了每种现有机制，看它能否**独立**（不组合、不做复杂配置）解决其中任一问题：

### 针对问题 1（智能编排：不跳步、不死板）

| 机制 | 能否独立解决 | 原因 |
| --- | --- | --- |
| Handoff 链 | 不能 | 只管流转顺序，不管"聚焦业务目标"也不管"灵活执行"（见下方详述） |
| 单条 Rule | 不能 | 原则能引导思维方式，但没有具体的进度追踪能力，无法保证不跳步 |
| 单个 Skill | 不能 | 要么太抽象没效果，要么太具体变成另一层死板流程 |
| TodoWrite（agent 内置待办） | 不能 | 仅存在于 agent 内存，对话被总结后可能丢失 |
| Cursor 内置功能 | 无此功能 | — |

### 针对问题 2（核心目标不被淹没）

| 机制 | 能否独立解决 | 原因 |
| --- | --- | --- |
| Notepad（记事本） | 不能 | 需要每次手动 @ 引用，忘了就失效 |
| alwaysApply Rule | 不能 | Rule 是静态的，不能承载每次任务不同的动态目标 |
| TodoWrite（agent 内部待办） | 不能 | 仅存在于 agent 内存，对话被总结后可能丢失，不持久化到磁盘 |
| 会话置顶/固定消息 | Cursor 无此功能 | — |

**结论：两个问题都没有单一的开箱即用机制能独立解决。**

### 关于 Handoff 链

Cursor 的命令支持 handoff 机制——在命令的 YAML 头部定义 `handoffs`，命令完成后 Cursor 会在对话底部显示按钮，点击即可触发下一个命令。设置 `send: true` 可以让下一步自动执行，无需用户点击。

Handoff 链看上去是编排多命令的天然方案，但它不能解决我们的核心问题：

- **不解决"死板"问题**：handoff 只是把手动输入命令名变成了点按钮或自动触发，agent 的执行方式没有任何变化
- **不解决"智能"问题**：agent 不会因为命令是通过 handoff 触发的就变得更聚焦业务目标
- **不通用**：handoff 只对特定命令组有效，无法覆盖 skill、MCP 或 specKit 等你没有修改权限的命令组
- **不解决注意力分散**：handoff 不涉及核心目标的维护，长对话中约束依然会被遗忘

Handoff 链更适合解决"使用便利性"问题（不需要记命令名和顺序），而非"执行质量"问题。它可以和最终方案搭配使用，但不能替代。

## 方案探索与技术选型

### 方案一：开发一个 Skill 来编排工作流

最初的想法是开发一个"工作流指挥"skill，教 agent 按照"发现工具 → 制定计划 → 逐步执行 → 验证完成"的流程工作。

问题：**skill 本身也可能被 agent 死板地执行**。我们希望解决死板执行的问题，结果可能引入一个新的死板流程——skill 自身变成了另一层机械指令。

此外，skill 通过 description 关键词触发，如果对话中没有命中触发词，skill 不会被加载。对于一个应该"始终生效"的机制，这个触发方式不够可靠。

| | |
| --- | --- |
| 约束力 | 中（第 6 层，被动触发） |
| 预期合规率 | ~70%（触发不可靠 + 内容可能被机械执行） |
| 优点 | 可以承载较长的指导内容 |
| 缺点 | 触发不可靠；内容本身可能被机械执行；不解决注意力分散 |

### 方案二：用 Cursor Rule 承载通用原则

用一条 `alwaysApply: true` 的 Rule，写入"聚焦业务目标"、"不跳过步骤"、"主动利用 MCP 和 skill"等原则。

问题：Rule 是静态的，可以承载不变的原则，但**无法承载每次任务不同的动态目标**。"为还款域创建监控大盘"和"生成系统设计文档"的核心目标完全不同，一条静态 Rule 覆盖不了。

单独使用 Rule 只能解决"思维方式引导"，不能解决"具体目标持久化"。

| | |
| --- | --- |
| 约束力 | 强（第 3 层，始终加载） |
| 预期合规率 | ~85%（始终加载，权重高，但抽象原则有理解偏差风险） |
| 优点 | 全局生效，不依赖触发 |
| 缺点 | 静态内容，无法承载动态目标；单独使用不解决任一问题 |

### 方案三：在每个命令文件中重复核心原则（Prompt Repetition）

在每个命令文件的开头都加上几行关键原则和"读取核心目标文件"的指令。这利用了 Prompt Repetition 技术——重复出现的指令有更高的注意力权重。

问题：**需要修改每一个命令文件**。自己的命令可以改，但 specKit 等第三方命令组没有修改权限。而且每新增一个命令都需要记得加上这段内容，维护成本高。

| | |
| --- | --- |
| 约束力 | 最高（指令在执行点的最近位置） |
| 预期合规率 | ~95%（位置最优 + 重复强化） |
| 优点 | 执行时权重最高，合规率最好 |
| 缺点 | 不通用，需要逐个修改命令文件，无法覆盖没有修改权限的命令 |

### 方案四（最终方案）：持久化核心目标文件 + alwaysApply Rule

如果能让每个任务的核心目标持久化到一个文件中，并用一条 alwaysApply Rule 让 agent 每次行动前都读取这个文件，就同时解决了两个问题：

- Rule 提供了始终生效的通用原则和"读取文件"的机械指令
- 文件提供了动态的、任务级别的核心目标和进度追踪
- 不需要修改任何命令文件——Rule 是全局生效的

| | |
| --- | --- |
| 约束力 | 强（Rule 在第 3 层 + 文件在每次行动前被刷新到最近位置） |
| 预期合规率 | 85-95%（见下方详细分析） |
| 优点 | 通用、不侵入命令文件、动态目标 + 静态原则互补 |
| 缺点 | 两个组件配合使用，依赖 agent 合规性读取文件 |

**选择方案四的原因**：它是所有方案中唯一同时满足"通用"（不改命令文件）、"动态"（每次任务不同的目标）和"可靠"（alwaysApply 的高优先级）三个要求的组合。而且它是最小组合——只有两个组件，没有多余的东西。

## 关键洞察：问题分解与方案收敛

回过头来看，"通用工作流指挥"要解决的其实是三个子问题：

| 子问题 | 含义 |
| --- | --- |
| A. 目标始终清晰 | agent 在整个过程中不忘记业务目标 |
| B. 步骤不被跳过 | 每一步都执行，产出都保留 |
| C. 工具被主动利用 | agent 会主动发现和使用 MCP、skill 等可用资源 |

如果"核心目标持久化"问题被解决了，那 A 直接被解决——agent 始终知道自己在做什么、为什么做。

而 A 解决之后，B 和 C 其实变得很轻：

- **B（不跳步）**：核心目标文件里可以包含一个"必须完成的步骤 checklist"。agent 每次读到核心目标时，也会看到这个 checklist，自然知道哪些还没做。
- **C（主动用工具）**：一条简短的 alwaysApply Rule 就够了："执行任务时，主动发现可用的 MCP 和 skill，用它们来丰富上下文和产出质量。"

换句话说——如果核心目标持久化能做到，就不需要单独的"工作流指挥 skill"了。"指挥"这个职能被分解成了：

- 核心目标文件（承载"做什么"和"做到哪了"）
- 一条 Rule（承载"怎么做"的原则）

这比一个独立的 conductor skill 更好，因为：

1. 没有额外的流程让 agent 机械执行
2. 核心目标是动态的、任务级别的，不是静态的方法论
3. 原则足够短，不容易被注意力稀释

所以关键路径变成了：**先解决核心目标持久化的问题。它是其他一切的基础。**

## 各指令类型的合规率预期

所有方案最终都依赖 agent 的合规性。以下是方案四中不同类型指令的预期合规率及原因：

| 指令类型 | 预期合规率 | 原因 |
| --- | --- | --- |
| "读取某个文件"（机械操作） | ~95% | 简单明确，不需要判断 |
| "建议用户做某事"（输出动作） | ~90% | agent 擅长"多说一句话"，只需生成文本 |
| "更新 checklist"（机械操作） | ~90% | 明确的文件修改操作 |
| "创建 session 文件"（需判断时机） | ~85% | 需要识别"多步骤任务"，有一定判断成分 |
| "不跳过步骤"（自律约束） | ~85% | 要求 agent 持续抑制自身行为，偶尔会失效 |
| "聚焦业务目标"（抽象原则） | ~80% | 抽象指令，agent 理解可能有偏差 |

对比没有任何机制时：核心约束在长对话中的存活率可能低于 50%。这套机制将关键指令的合规率提升到了 80-95% 的区间。

## 最终方案详述

两个组件，没有多余的东西。

### 组件一：会话目标文件（`.cursor/sessions/objective.md`）

这个文件由 agent 在识别到多步骤任务时自动创建，内容包括：

```markdown
## 核心目标
[一句话描述业务目标]

## 执行进度
- [x] 已完成的步骤
- [ ] 未完成的步骤
- ...

## 关键约束
- [约束1]
- [约束2]
```

- 持久化到磁盘（不是内存，对话被总结后也不会丢）
- 不进 git（`.cursor/sessions/` 目录应被 .gitignore 忽略）
- agent 每完成一步自动勾选 checkbox，发现新约束时自动追加

### 组件二：alwaysApply Rule

一条全局规则，安装到 `~/.cursor/rules/session-objective.md`，始终加载。它告诉 agent：

**读取**：如果 objective.md 存在，每次行动前读取它。

**创建**：识别到多步骤任务且文件不存在时，与用户确认目标后创建。首次创建时向用户简要解释这个机制。

**更新**：完成步骤后勾选 checkbox，发现新约束时追加。不自行修改核心目标，只有用户确认后才能改。

**主动建议**：
- 用户强调某个要求时 → 建议加进核心目标
- 发现某个约束与实际矛盾时 → 建议更新
- 用户改变方向时 → 建议更新目标
- checklist 全部完成时 → 提示归档或清除
- 当前对话与目标无关时 → 提示是否更新或清除，避免旧目标干扰新任务

**执行原则**：
- 不能自行跳过 checklist 中未完成的步骤
- 主动发现和利用可用的 MCP、skill 丰富产出
- 如果认为某步骤不需要，必须向用户说明并获得确认

## 为什么这套机制有效

### 解决问题 1（不跳步、不死板）

agent 每次行动前读到的不是"接下来执行某条命令"这样的指令，而是"为还款域创建监控大盘"这样的业务目标——思考锚点是"怎样达成目标"，不是"下一条命令是什么"。

同时，checklist 让未完成的步骤始终可见。agent 参与了这个 checklist 的创建，知道每一步的存在意义，不会像外部强加的流程那样被机械执行。

### 解决问题 2（注意力不分散）

关键约束写在 objective.md 里，不再散落在对话历史中。每次行动前重新读取文件，等效于每轮对话都自动"呐喊"了一次——但不需要用户手动重复。

当用户在对话中提出新约束时，agent 会建议写入 objective.md。一旦写入，这条约束就脱离了对话历史的注意力衰减曲线，成为每次都会被读取的持久化内容。

**本质上，objective.md 是一个用"文件 + 每次读取"模拟出来的"置顶消息"。** Cursor 没有原生的会话置顶功能，这是目前能做到的最接近的效果。

## 适用场景

这套机制不依赖 specKit、命令组或任何特定工具栈。只要满足以下任一条件，它就能发挥作用：

**任何超过 10 轮的对话（解决问题 2）**

注意力分散是大模型的固有特性，跟你用什么工具无关。典型场景：

- 用几个 skill 做复杂重构，第 3 轮说的"不要改公共接口签名"到第 12 轮被忘了
- 长时间 debug 会话，最初的根因假设被后续排查信息淹没
- Code review 时，开头提的三个关注点到后面只记得一个

这些场景里 objective.md 的"关键约束"部分能把关键要求持久化，每轮自动刷新到 agent 的注意力范围内。

**任何多步骤任务（解决问题 1）**

步骤不一定是命令，任何需要多轮协作才能完成的任务都算：

- "帮我设计一个缓存方案" → 分析现状、设计方案、评估取舍、写技术文档，agent 可能跳过"评估取舍"直接给方案
- "重构这个模块" → 梳理依赖、改代码、改测试、验证 CI，agent 可能改完代码就宣布完成
- "调研三个开源方案" → 逐一分析优缺点然后对比，agent 可能只看了两个就下结论

checklist 在这些场景中的价值一样大——agent 参与了步骤拆解，知道每一步为什么存在，不会随意跳过。

**与 specKit / 命令组搭配时的区别**

有命令组时，几乎每次使用都会触发 objective.md 的创建（命令组天然是多步骤的）。没有它们时，Rule 需要 agent 自行判断"这是一个多步骤任务"才会触发——这个判断的合规率约 85%，偶尔会漏掉。但只要手动说一句"帮我建个核心目标"，就能正常运转。

## 约束与局限

这套机制的每一环都依赖 agent 的合规性。alwaysApply Rule 在用户可控的机制中约束力最高，但它最终还是 prompt 的一部分，通过注意力机制影响 agent 行为，不是程序层面的硬约束。

比没有任何机制时好得多，但不是 100% 保证。在 Cursor 提供原生的会话级置顶或强制上下文注入功能之前，这是当前平台能力下的最优解。

## 安装

这里的安装方式基于公司内部的分发机制，对外不具有参考性

```bash
# 克隆仓库（首次）
git clone <repo-url> ~/feishu-skill

# 安装
~/feishu-skill/scripts/install.sh workflow-conductor

# 更新
cd ~/feishu-skill && git pull
~/feishu-skill/scripts/install.sh workflow-conductor --force
```

安装后无需任何配置。下一个新会话开始时，agent 会自动加载这条规则。

## 关闭与恢复

如果暂时不需要这个功能，编辑 `~/.cursor/rules/session-objective.md`，把头部的 `alwaysApply: true` 改成 `alwaysApply: false`。Rule 文件还在但不再自动加载，等于全局静默。想恢复时改回 `true`，新开一个会话即可生效。

如果当前有活跃的 objective.md，关闭 Rule 后它不会被自动清理，但也不会再被读取——下次重新启用时，agent 会发现这个文件并提示你是否继续或清除。
